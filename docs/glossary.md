# Agent Role Play 용어 사전

프로젝트에서 사용하는 핵심 용어와 개념을 정의합니다.

## 프로젝트 관련 용어

### CareerMap (커리어맵)

- **정의**: 이력 기반 직장 추천 및 동료 네트워크 분석 서비스
- **표기 규칙**:
  - 문서/설명: "CareerMap" 또는 "커리어맵" 사용
  - 코드/API: "CareerMap" 사용 (PascalCase)
  - URL/경로: "career-map" 사용 (kebab-case)

### 에이전트 (Agent)

- **정의**: 특정 역할(PM, 프론트엔드, 서버, 문서화)을 담당하는 자동화된 작업 주체
- **표기 규칙**:
  - 한국어 문서: "에이전트"
  - 영문 문서/코드: "Agent"
- **종류**:
  - PM 에이전트: 업무 지시 및 품질 관리
  - 프론트엔드 에이전트: Flutter 앱 개발
  - 서버 에이전트: Kotlin & Spring 백엔드 개발
  - 문서화 에이전트: 문서 정리 및 정합성 유지

## 문서 관련 용어

### 부트스트랩 파일 (Bootstrap File)

- **정의**: 에이전트가 작업 시작 시 가장 먼저 읽어야 하는 기본 문서
- **포함 내용**: 역할 정의, 프로젝트 맥락, 현재 상태, 우선순위
- **예시**:
  - PM: `docs/career_map_plan_doc.md`, `docs/roles/pm.md`
  - 프론트엔드: `docs/roles/frontend.md`, `docs/career_map_story_board.md`
  - 서버: `docs/roles/server.md`, `docs/career_map_erd.md`
  - 문서화: `docs/roles/documentation.md`

### 스레드 (Thread)

- **정의**: 특정 업무/이슈에 대한 시간순 협업 기록 파일
- **위치**: `docs/threads/` 디렉토리
- **네이밍 규칙**: `YYYY-MM-DD-주제.md` (예: `2025-12-19-profile-card.md`)
- **구성 요소**:
  - `[지시]`: 업무 지시서
  - `[업데이트]`: 진행 상황
  - `[결과]`: 작업 결과 보고
  - `[결정]`: 의사결정 기록
  - `[이슈]`: 문제/리스크 기록

### 템플릿 (Template)

- **정의**: 표준화된 문서 양식
- **위치**: `docs/templates/` 디렉토리
- **종류**:
  - `task-brief.md`: 업무 지시서
  - `task-report.md`: 업무 결과 보고서
  - `issue-log.md`: 이슈/리스크 기록
  - `decision-record.md`: 의사결정 기록
  - `plan.md`: 주간/스프린트 계획

## 업무 프로세스 용어

### 완료 조건 (Completion Criteria)

- **정의**: 업무가 성공적으로 완료되었다고 판단하는 명확한 기준
- **표기 규칙**: "완료 조건"으로 통일 ("완료 기준"은 사용 지양)
- **포함 요소**:
  - 기능적 요구사항 충족
  - 테스트 통과
  - 코드 리뷰 승인
  - 문서화 완료

### 산출물 (Deliverable)

- **정의**: 작업 결과로 제출해야 하는 구체적인 결과물
- **예시**: PR 링크, 스토리북 스토리, API 문서, 테스트 결과

### 지시서 (Task Brief)

- **정의**: PM 에이전트가 작성하는 업무 요청 문서
- **필수 항목**: 목표, 배경, 요구사항, 범위, 산출물, 완료 조건, 마감일

### 결과 보고서 (Task Report)

- **정의**: 작업 완료 후 에이전트가 작성하는 결과 문서
- **필수 항목**: 작업 요약, 변경 내용, 테스트 결과, 이슈, 다음 작업 제안

### 검수 (Review)

- **정의**: PM/문서화 에이전트가 결과물의 품질을 확인하는 프로세스
- **체크 항목**: 요구사항 매핑, 품질 근거, 리스크 처리, 결정 기록

## 기술 스택 용어

### 서브모듈 (Submodule)

- **정의**: Git 서브모듈을 통해 문서 레포를 코드 레포에 연결하는 전략
- **목적**: 문서와 코드를 분리 관리하면서 단일 소스 유지
- **경로**: 코드 레포의 `docs/` 디렉토리

### MVP (Minimum Viable Product)

- **정의**: 최소 기능 제품, 핵심 가치를 검증할 수 있는 최소 범위
- **CareerMap MVP 범위**:
  - 이메일 회원가입/로그인
  - 이력 입력
  - 직장 추천
  - 동료 추가 및 네트워크 예측

## 아키텍처 용어

### Clean Architecture

- **정의**: 프론트엔드에서 사용하는 아키텍처 패턴
- **계층**: Presentation - Domain - Data
- **참고**: https://github.com/KoreanTuna?tab=repositories

### 상태관리 (State Management)

- **프론트엔드**: flutter_hooks + hooks_riverpod
- **정의**: 앱 내 데이터 흐름과 UI 상태를 관리하는 방식

## 프라이버시/보안 용어

### 공개 범위 (Visibility)

- **정의**: 사용자 정보를 누구에게 공개할지 제어하는 설정
- **옵션**:
  - 전체 공개 (PUBLIC)
  - 친구만 (FRIENDS)
  - 비공개 (PRIVATE)

### 마스킹 (Masking)

- **정의**: 민감 정보(예: 회사명)를 숨기고 일반적 카테고리로 표시
- **예시**: "썸하우컴퍼니" → "스타트업"

## 품질 관련 용어

### 이력 완성도 (Resume Completion)

- **정의**: 사용자 프로필의 작성 정도를 0~100% 점수로 표시
- **목적**: 추천 정확도 향상 유도

### 적합도 점수 (Fit Score)

- **정의**: 사용자 이력과 추천 직무의 매칭도를 0~100 점수로 표시
- **기준**: 스킬 매칭, 도메인 경험, 경력연차, 희망조건 일치도

### 갭 분석 (Gap Analysis)

- **정의**: 추천 직무 요구사항과 사용자 현재 역량 간 차이 분석
- **산출물**: 부족 스킬/경험 목록 + 준비 액션 제안

## 협업 규칙 용어

### 차단 이슈 (Blocking Issue)

- **정의**: 작업 진행을 막는 심각한 문제
- **처리**: `issue-log` 템플릿으로 즉시 공유, @오너 호출

### 승인 라벨 (Approval Label)

- **정의**: 스레드 파일에 남기는 작업 상태 표시
- **종류**: "승인됨", "보완필요", "블록됨"

### 스탠드업 (Stand-up)

- **정의**: 짧은 일일 진행 상황 공유
- **형식**: `[날짜] 진행/차단/다음 액션` 3줄

## 버전 관리 용어

### PR (Pull Request)

- **정의**: 코드 변경사항을 메인 브랜치에 병합 요청하는 프로세스
- **필수 요소**:
  - PR 설명
  - 관련 docs 스레드 링크
  - 테스트 결과

### 커밋 메시지 (Commit Message)

- **형식**:

  ```
  요약 (1줄)

  상세 설명 (선택)

  🤖 Generated with Claude Code

  Co-Authored-By: Claude Sonnet 4.5 <noreply@anthropic.com>
  ```
